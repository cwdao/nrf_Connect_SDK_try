# BLE CS 本地 IQ 提取方案工作计划

## 一、项目目标

当前示例程序的完整链路是：

- Initiator 通过 HCI 获取本地 CS subevent report；
- 再通过 RAS 从对端获取 peer ranging data；
- 将本地与对端数据合并后生成 `cs_de_report`；
- 最终计算距离。

本阶段拟开展的工作是：

> **从现有双端测距流程中裁剪出“仅提取本地 IQ”的处理链路，直接基于本地 HCI subevent report 中的 PCT，解析得到本地 IQ 数据，不再依赖 RAS 获取对端数据，也不再进行距离计算。**

最终希望达到的效果是：

- BLE CS procedure 可持续重复运行；
- 每次 subevent 到来时，能够直接从本地 HCI report 中提取 PCT；
- 将 PCT 转换为本地 IQ；
- 支持连续输出、缓存或存储本地 IQ 数据；
- 为后续开展本地信道特征分析、PCT统计分析或自定义算法研究提供基础数据通路。

------

# 二、总体技术路线

本次改造的总体技术路线如下：

1. **保留 BLE 连接与 CS procedure 配置流程**
   - 包括扫描、连接、安全、CS capability exchange、config create、procedure enable。
2. **删除或旁路 RAS 数据获取链路**
   - 不再订阅 `RD_READY` / `RD_OVERWRITTEN` / `On-demand RD` / `RAS CP`。
   - 不再等待对端提供 peer ranging data。
3. **将数据处理入口前移到 `subevent_result_cb()`**
   - 直接在本地 HCI subevent 回调中读取 `step_data_buf`。
4. **实现本地 step data 解析**
   - 遍历 step；
   - 筛选 mode 2 / mode 3；
   - 提取 `tone_info[].phase_correction_term`；
   - 通过 PCT 解码得到本地 IQ。
5. **建立本地 IQ 数据输出链路**
   - 支持日志打印、结构化缓存、串口导出或 Flash 存储。

# 三、具体工作计划

------

## 工作一：梳理现有 BLE CS 测距数据链路

### 工作内容

对当前工程代码进行结构梳理，明确现有双端测距流程中各模块的职责边界，重点定位：

- 本地 HCI subevent report 的来源与回调入口；
- RAS 对端数据获取流程；
- `cs_de_report` 的生成路径；
- PCT 到 IQ 的解析位置；
- 距离计算与本地 IQ 提取的耦合关系。

### 重点分析对象

- `main.c`
- `cs.c`
- `ras_rreq.c`
- `cs_de.c`

### 输出物

- 一张当前系统数据流转图；
- 一份“本地数据链路 / 对端数据链路 / 距离计算链路”的拆分说明；
- 明确本地 IQ 所依赖的最小功能集合。

### 目的

确保后续修改时不会误删 BLE CS 必要流程，也避免将“本地 PCT 解析”与“对端 RAS 取数”混淆。

------

## 工作二：确认本地 IQ 的最小数据来源与解析规则

### 工作内容

验证以下问题并形成结论：

1. 本地 IQ 是否可仅依赖 HCI 上报获得；
2. 本地 IQ 是否必须等待 peer 数据对齐后才能解析；
3. PCT 的编码格式及其到 IQ 的转换方法；
4. mode 2 / mode 3 下本地 tone info 的结构与读取方式；
5. antenna path 与 permutation index 的映射关系。

### 关键技术点

- `bt_le_cs_parse_pct()`
- `bt_le_cs_step_data_parse()`
- `bt_le_cs_get_antenna_path()`

### 输出物

- PCT→IQ 解析说明文档；
- 本地 IQ 字段映射关系表；
- 单个 step 数据结构解析说明。

### 目的

为后续应用层直接解析本地 IQ 提供理论和代码依据。

------

## 工作三：设计“仅本地 IQ 提取”新流程

### 工作内容

在现有工程基础上设计新的处理链路，将数据处理逻辑从：

- “收集本地 step → 等待 RAS → 合并本地/对端 → 计算距离”

调整为：

- “收到本地 subevent → 直接解析 PCT → 生成本地 IQ”。

### 拟调整后的流程

- 保留 BLE CS procedure 周期运行；
- 保留 HCI subevent 回调；
- 在 `subevent_result_cb()` 中直接解析本地 step data；
- 移除对 `latest_peer_steps`、`ranging_data_ready_cb()`、`ranging_data_get_complete_cb()` 的依赖。

### 输出物

- 一张改造后目标流程图；
- 一份新旧流程差异对比说明。

### 目的

形成明确的软件架构调整方案，为代码修改提供依据。

------

## 工作四：裁剪 RAS 相关代码依赖

### 工作内容

从应用层剥离“仅用于获取对端数据”的相关逻辑，主要包括：

- 去除 RAS 的订阅和回调注册；
- 去除 `latest_peer_steps` 缓冲区；
- 去除 `ranging_data_ready_cb()` / `ranging_data_overwritten_cb()` / `ranging_data_get_complete_cb()`；
- 去除等待 peer data 的同步控制逻辑；
- 去除 `cs_de_populate_report()` 和 `cs_de_calc()` 的调用。

### 需重点评估

- 是否还需要 RAS service discovery；
- 哪些 discovery 流程可一并删除；
- 是否会影响 BLE CS 配置与 procedure enable。

### 输出物

- 一份待删除/待保留模块清单；
- 一版精简后的应用层结构草案。

### 目的

降低系统复杂度，使程序聚焦于“本地 IQ 提取”目标。

------

## 工作五：实现本地 step data 解析模块

### 工作内容

新增或重构应用层解析函数，实现以下能力：

1. 从 `subevent_result_cb()` 获取 `step_data_buf`；
2. 遍历 subevent 中的每个 step；
3. 判断 step mode；
4. 读取 `tone_info[].phase_correction_term`；
5. 将 PCT 解析为 `I/Q`；
6. 将本地 IQ 按 channel、antenna path、step 等维度组织输出。

### 建议实现内容

- 一个 `parse_local_iq_from_subevent()` 函数；
- 一个 `local_step_iq_cb()` step 处理函数；
- 一个 `parse_pct_local()` 或复用现有 `bt_le_cs_parse_pct()` 的本地接口。

### 输出物

- 本地 IQ 解析代码模块；
- 本地 IQ 数据结构定义；
- 每个 subevent 的本地 IQ 提取结果。

### 目的

建立本项目的核心能力：**持续从本地 CS subevent 中直接提取 IQ**。

# 六、风险与难点

汇报时建议也带上这部分，会显得考虑比较完整。

------

## 1. 天线路径映射复杂

本地 tone info 的读取可能受 antenna permutation index 影响，不能简单按数组下标直接解释。

## 2. `n_ap` 来源需重新确认

原先部分信息来自 peer ranging header，去掉 RAS 后需要重新确定本地可用 antenna path 数的来源。

## 3. 高频日志可能成为瓶颈

连续 subevent 下，如果直接打印每个 IQ，系统可能出现性能问题，需要缓存或二进制输出。

## 4. 回调中处理开销需受控

`subevent_result_cb()` 是高频入口，解析和存储逻辑需足够轻量，避免影响实时性。

## 5. 现有工程与测距算法耦合较深

需要小心拆分，防止误删影响 BLE CS procedure 正常运行的必要代码。
